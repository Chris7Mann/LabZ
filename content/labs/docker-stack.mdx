## Guida tecnica alla creazione di uno stack multi-container con Docker Compose.

In questo laboratorio realizzeremo un ambiente composto da un Reverse Proxy Nginx e una Web Application interna, separati tramite rete Docker dedicata e con persistenza dei dati tramite Docker Volume.



<SectionTitle>Scenario del Laboratorio</SectionTitle>

<Text>

Immaginiamo di dover pubblicare una piccola applicazione web interna per un blog tecnico di informatica e networking.

L'applicazione non deve essere esposta direttamente verso l'esterno.

Tutte le richieste HTTP dovranno passare prima da Nginx, che fungerà da Reverse Proxy.

I dati dell'applicazione dovranno essere persistenti, quindi non dovranno andare persi quando un container viene eliminato o ricreato.

</Text>

<Info>

Questo laboratorio introduce concetti fondamentali utilizzati anche in ambienti professionali: isolamento dei servizi, reverse proxy, Docker networking, service discovery e persistenza dei dati.

</Info>



<SectionTitle>Architettura Finale</SectionTitle>

<Text>

L'architettura finale sarà composta da due container principali:

* **blog-nginx**: Reverse Proxy Nginx esposto sulla porta 80 dell'host
* **blog-app**: applicazione web interna basata su Apache HTTP Server

I due container comunicheranno tramite una rete Docker privata chiamata **blog-network**.

I dati persistenti saranno salvati nel volume Docker **blog-data**.

</Text>

<Diagram code="
graph TD
    Client[Client Browser] --> Host[Host Docker Porta 80]
    Host --> Nginx[Container blog-nginx]
    Nginx --> Network[Docker Network blog-network]
    Network --> App[Container blog-app]
    App --> Volume[Docker Volume blog-data]
"/>

<Warning>

Il container applicativo **blog-app** non verrà esposto direttamente sull'host.

L'unico servizio pubblicato verso l'esterno sarà **Nginx**.

</Warning>

<StepCard label="B" title="Creazione della cartella di progetto">

Creiamo una directory dedicata al laboratorio.

<Terminal code="mkdir docker-blog-lab
cd docker-blog-lab"/>

Creiamo poi le sottocartelle necessarie.

<Terminal code="mkdir app

La struttura del progetto sarà questa:

<Terminal code="docker-blog-lab/

│

 └── default.conf"/>

<Note>

**Perché separare le cartelle?**

Separare configurazione Nginx, codice applicativo e file Compose rende il progetto più leggibile, più ordinato e più simile a una struttura professionale.

</Note>

</StepCard>

<StepCard label="D" title="Configurazione di Nginx come Reverse Proxy">

Creiamo il file di configurazione di Nginx.

<Terminal code="nano nginx/default.conf"/>

Inseriamo questa configurazione:

<Terminal code="server {

```
listen 80;

location / {

    proxy_pass http://blog-app:80;

    proxy_set_header Host $host;

    proxy_set_header X-Real-IP $remote_addr;

    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

}
```

}"/>

<Note>

**proxy_pass http://blog-app:80**

Questa direttiva dice a Nginx di inoltrare le richieste HTTP verso il servizio chiamato **blog-app** sulla porta 80.

Non stiamo usando un indirizzo IP statico.

Docker Compose fornisce un DNS interno che permette ai container di risolversi tramite il nome del servizio.

</Note>

<Warning>

Non usare **localhost** dentro `proxy_pass`.

Dentro un container, `localhost` indica il container stesso, non il tuo computer e non un altro container.

Per comunicare con un altro servizio Docker Compose devi usare il nome del servizio, in questo caso **blog-app**.

</Warning>

</StepCard>



<StepCard label="E" title="Creazione del file docker-compose.yml">

Creiamo il file principale che descrive lo stack.

<Terminal code="nano docker-compose.yml"/>

Inseriamo questa configurazione:

<Terminal code="services:

blog-app:

```
image: httpd:2.4

container_name: blog-app

volumes:
  - ./app:/usr/local/apache2/htdocs
  - blog-data:/data

networks:
  - blog-network
```

nginx:

```
image: nginx:1.27

container_name: blog-nginx

ports:
  - &quot;80:80&quot;

volumes:
  - ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro

depends_on:
  - blog-app

networks:
  - blog-network
```

volumes:

blog-data:

networks:

blog-network:
driver: bridge"/>

</StepCard>



<StepCard label="F" title="Analisi del servizio blog-app">

Analizziamo il primo servizio dello stack.

<Terminal code="blog-app:

image: httpd:2.4

container_name: blog-app"/>

Il servizio **blog-app** utilizza l'immagine ufficiale di Apache HTTP Server.

Il container viene chiamato esplicitamente **blog-app**.

<Terminal code="volumes:

* ./app:/usr/local/apache2/htdocs
* blog-data:/data"/>

Il primo mount collega la cartella locale **app** alla directory usata da Apache per servire i file web.

Il secondo mount collega il volume Docker **blog-data** alla directory **/data** dentro il container.

<Info>

In questo lab usiamo sia un bind mount sia un volume Docker:

* **Bind mount**: collega una cartella locale del progetto al container
* **Docker Volume**: spazio gestito direttamente da Docker per conservare dati persistenti

</Info>

</StepCard>



<StepCard label="G" title="Analisi del servizio nginx">

Analizziamo il servizio Nginx.

<Terminal code="nginx:

image: nginx:1.27

container_name: blog-nginx

ports:
- &quot;80:80&quot; "/>

La direttiva **ports** pubblica la porta 80 del container sulla porta 80 dell'host.

Questo significa che potremo raggiungere Nginx dal browser usando:

<Terminal code="http://localhost"/>

Analizziamo il mount della configurazione.

<Terminal code="volumes:

* ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro"/>

Il file locale **nginx/default.conf** viene montato dentro il container Nginx.

L'opzione **:ro** significa **read-only**.

<Warning>

Montare la configurazione Nginx in sola lettura è una buona pratica.

Il container può leggere il file, ma non può modificarlo.

</Warning>

</StepCard>



<StepCard label="H" title="Analisi della rete Docker">

Nel file Compose abbiamo definito una rete dedicata.

<Terminal code="networks:

blog-network:
driver: bridge"/>

Entrambi i servizi sono collegati alla stessa rete.

<Terminal code="networks:

* blog-network"/>

Questo permette a **blog-nginx** e **blog-app** di comunicare tra loro.

<Info>

Docker Compose crea automaticamente un sistema DNS interno.

Per questo motivo Nginx può raggiungere Apache usando il nome:

**blog-app**

</Info>

</StepCard>

<StepCard label="L" title="Verifica DNS interno tra container">

Entriamo nel container Nginx.

<Note>

**getent hosts blog-app** verifica se il nome **blog-app** viene risolto in un indirizzo IP.

Questo dimostra che il DNS interno di Docker Compose sta funzionando.

</Note>

</StepCard>




<StepCard label="M" title="Test della raggiungibilità applicativa interna">

Sempre dal container Nginx, possiamo testare la raggiungibilità del servizio applicativo.

<Terminal code="curl http://blog-app:80"/>

Dovremmo ricevere il contenuto HTML della pagina.

<Warning>

Se il comando **curl** non è disponibile dentro il container Nginx, non è un errore della tua configurazione.

Alcune immagini Docker sono minimali e non includono tutti gli strumenti di troubleshooting.

</Warning>

</StepCard>




<StepCard label="N" title="Verifica del volume Docker">

Visualizziamo i volumi Docker.

<Terminal code="docker volume ls"/>

Dovremmo trovare un volume simile a:

<Terminal code="docker-blog-lab_blog-data"/>

Ispezioniamo il volume.

<Terminal code="docker volume inspect docker-blog-lab_blog-data"/>

<Info>

Anche i volumi, come le reti, possono essere prefissati con il nome della directory del progetto.

</StepCard>




<StepCard label="O" title="Scrittura di un dato persistente nel volume">

Entriamo nel container applicativo.

<Terminal code="docker exec -it blog-app bash"/>

Creiamo un file dentro la directory montata sul volume Docker.

<Terminal code="echo &quot;Dato persistente salvato nel volume Docker&quot; &gt; /data/persistenza.txt"/>

Verifichiamo il contenuto del file.

<Terminal code="cat /data/persistenza.txt"/>

Output atteso:

<Terminal code="Dato persistente salvato nel volume Docker"/>

</StepCard>




<StepCard label="P" title="Test reale di persistenza">

Usciamo dal container.

<Terminal code="exit"/>

Fermiamo ed eliminiamo i container dello stack.

<Terminal code="docker compose down"/>

Riavviamo lo stack.

<Terminal code="docker compose up -d"/>

Rientriamo nel container applicativo.

<Terminal code="docker exec -it blog-app bash"/>

Controlliamo se il file è ancora presente.

<Terminal code="cat /data/persistenza.txt"/>

Se il file esiste ancora, la persistenza tramite Docker Volume sta funzionando.

<Note>

**docker compose down** elimina i container e la rete creata da Compose, ma non elimina automaticamente i volumi.

Per eliminare anche i volumi bisogna usare un comando specifico.

</Note>

</StepCard>




<StepCard label="Q" title="Eliminazione completa dello stack">

Per fermare ed eliminare container e rete:

<Terminal code="docker compose down"/>

Per eliminare anche i volumi associati allo stack:

<Terminal code="docker compose down -v"/>

<Warning>

Il comando **docker compose down -v** elimina anche i volumi.

Se nel volume sono presenti dati importanti, verranno rimossi.

Usalo solo quando sei sicuro di voler cancellare anche i dati persistenti.

</Warning>

</StepCard>




<SectionTitle>Troubleshooting</SectionTitle>

<Text>

Questa sezione raccoglie i controlli più importanti.

</Text>




<StepCard label="R" title="Controllare lo stato dei container">

Visualizzare i container attivi:

<Terminal code="docker ps"/>

Visualizzare anche i container fermi:

<Terminal code="docker ps -a"/>

</StepCard>




<StepCard label="S" title="Controllare i log dei servizi">

Log del container Nginx:

<Terminal code="docker logs blog-nginx"/>

Log del container Apache:

<Terminal code="docker logs blog-app"/>

Seguire i log in tempo reale:

<Terminal code="docker logs -f blog-nginx"/>

<Info>

I log sono il primo punto da controllare quando un container non parte, va in crash o restituisce errori HTTP.

</Info>

</StepCard>




<StepCard label="T" title="Errore porta 80 già occupata">

Se ottieni un errore simile a:

<Terminal code="bind: address already in use"/>

significa che la porta 80 dell'host è già occupata da un altro processo.

Controlla quale processo la sta usando.

<Terminal code="ss -tulpn | grep :80"/>

Soluzione temporanea: modifica il mapping porta nel file Compose.

<Terminal code="ports:

* &quot;80:80&quot;"/>

Poi accedi dal browser con:

<Terminal code="http://localhost:8080"/>

</StepCard>




<StepCard label="U" title="Errore 502 Bad Gateway">

Se Nginx restituisce **502 Bad Gateway**, significa che il Reverse Proxy non riesce a raggiungere l'applicazione interna.

Controlla che il container applicativo sia attivo.

<Terminal code="docker ps"/>

Controlla i log di Nginx.

<Terminal code="docker logs blog-nginx"/>

Verifica che il nome nel file Nginx sia corretto.

<Terminal code="proxy_pass http://blog-app:80;"/>

<Warning>

Un errore comune è scrivere un nome servizio diverso da quello definito nel file **docker-compose.yml**.

Il nome usato in **proxy_pass** deve corrispondere al nome del servizio o al nome risolvibile nella rete Docker.

</Warning>

</StepCard>




<StepCard label="V" title="Verifica configurazione Nginx">

Entriamo nel container Nginx.

<Terminal code="docker exec -it blog-nginx bash"/>

Verifichiamo la configurazione.

<Terminal code="nginx -t"/>

Output atteso:

<Terminal code="nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful"/>

</StepCard>




<SectionTitle>Best Practice di Produzione</SectionTitle>

<Warning>

In produzione evita il tag **latest**.

Usare versioni specifiche come **nginx:1.27** o **httpd:2.4** rende il deployment più prevedibile.

</Warning>

<Warning>

Non esporre direttamente il container applicativo se non è necessario.

In questo lab esponiamo solo Nginx e manteniamo l'applicazione interna raggiungibile solo dalla rete Docker.

</Warning>

<Info>

Un Reverse Proxy può essere usato anche per gestire:

* Certificati HTTPS
* Redirect HTTP verso HTTPS
* Compressione
* Header di sicurezza
* Rate limiting
* Logging centralizzato
* Routing verso più applicazioni

</Info>

<Note>

**Principio chiave:**

Container = processo effimero

Volume = dato persistente

Network = comunicazione controllata

Reverse Proxy = punto di ingresso

</Note>
